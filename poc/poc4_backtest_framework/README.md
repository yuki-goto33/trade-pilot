# PoC-4: 日本株バックテスト基盤の選定・検証

MT4 の代替として、Python 系バックテストフレームワーク + J-Quants 過去 2 年日足で
「シグナルに従って売買した場合」の成績（勝率・累積損益・最大ドローダウン）を検証できるかを確認する。

- 検証日: 2026-07-04
- 環境: macOS (arm64) / Python 3.9.6 (`.venv`) / numpy 2.0.2 / pandas 2.3.3
- データ: `data/prices_jquants.csv`（PoC-1 取得。11 銘柄 × 2 年日足、2024-04-11〜2026-04-10、調整後 OHLCV あり）

**選定: Backtesting.py 0.6.5**

- 決め手: Python 3.9 の既存環境（numpy 2 系）に唯一クリーンに入り、かつメンテが活発な候補が Backtesting.py のみ。
  vectorbt は numpy ダウングレード必須で PoC-1〜3 と同一環境に共存できず除外。backtrader は保守停止で除外。
- 複数銘柄ネイティブ対応がない点は、銘柄ごとのループ + equity 合算で実用上問題ないことを確認（11 銘柄 × 2 年で 0.45 秒）。

## 2. 実行方法

```bash
cd trade-pilot

# 依存インストール（requirements.txt に backtesting を追加済み）
.venv/bin/pip install backtesting

# (a) データ確認
.venv/bin/python poc/poc4_backtest_framework/data_loader.py

# (b) ベースライン: ゴールデンクロス戦略 (SMA25/75)
.venv/bin/python poc/poc4_backtest_framework/run_golden_cross.py

# (c) ダミーシグナル生成（ゴールデンクロスから機械生成）
.venv/bin/python poc/poc4_backtest_framework/make_dummy_signals.py

# (d) シグナル駆動バックテスト
.venv/bin/python poc/poc4_backtest_framework/run_signals.py poc/poc4_backtest_framework/signals.csv
.venv/bin/python poc/poc4_backtest_framework/run_signals.py poc/poc4_backtest_framework/signals.csv --min-confidence 0.6
```

共通条件: 初期資金 1,000 万円/銘柄、手数料 0.1%（約定代金比）、単元株 100 株、
シグナル確定バーの翌営業日寄り付きで約定（`trade_on_close=False`）、現物ロングのみ。
スリッページは `spread` パラメータで設定可能（本検証では 0）。

### 3.2 シグナル駆動インターフェース

`signals.csv`（列: `date, code, action[buy/sell/hold], confidence`）を読むだけでバックテストできる
runner (`run_signals.py`) を実装し、以下を確認した。

- **等価性検証 OK**: ゴールデンクロスから機械生成したダミーシグナル（buy 32 / sell 36 / hold 517 行）を
  `run_signals.py` に与えた結果が、3.1 のベースラインと**銘柄別 trades・勝率・損益・最大 DD すべて完全一致**
  （`results/signal_backtest_results.csv` と `results/golden_cross_results.csv` の diff で確認）
- **hold の無視**: hold 517 行はポジションに影響しないことを確認
- **confidence 足切り**: `--min-confidence 0.6` で低確信度シグナルのみ除外されて動作
  （例: 40630 が 3 → 2 トレードに減少）
- **入力バリデーション**: 不正 action（例: `short`）、`date×code` 重複、必須列不足で明示的なエラー、
  価格データにない銘柄コードは warning + スキップ

シグナルの意味論（PoC-3 / 将来の LLM シグナルとの接続規約）:
`date` 行のシグナル = その日の引け時点で確定した判断 → **翌営業日の寄り付きで約定**。
「毎朝寄り付き前にレポート配信 → 当日寄りで執行」の運用は、前日日付のシグナルとして表現する。

PoC-3 の `signal_schema.json`（`signal`: buy/sell/hold, `confidence`: 0-100 整数）との対応:
`signal` → `action` はそのまま、`confidence` は **100 で割って 0.0-1.0 に正規化**して signals.csv に変換する。

## 5. 発見事項・制約

- **Python 3.9 起因の制約**:
  - vectorbt は 3.9 だと numpy<1.24 へのダウングレードが必須で、既存環境（numpy 2.0.2）と共存不可 → 除外の決定打。
  - Python 3.9 は 2025-10 に EOL 済み。Backtesting.py 0.6.5 は `requires-python >=3.9` で当面動くが、
    将来リリースで 3.10+ に引き上げられる可能性が高い。本格実装フェーズでは **Python 3.11+ への移行を推奨**
    （移行すれば vectorbt / vectorbt PRO も再び選択肢に入る）。
- **単元株制約は実際に効く**: キーエンス（68610、株価 5〜7.7 万円）は 1 単元 500〜770 万円。
  銘柄あたり資金 500 万円の設定では 1 単元も買えず取引ゼロになった（→ 本検証では 1,000 万円/銘柄に設定）。
  universe 選定（PoC-2）や実運用の資金設計で「単元価格 > 割当資金」の銘柄の扱いを決める必要がある。
- **Backtesting.py は 1 Backtest = 1 銘柄**: 銘柄間の資金融通（ポートフォリオ一体運用）は表現できない。
  本 PoC では銘柄別に資金を固定割当して equity 合算する方式とした。銘柄横断の資金管理・銘柄選択を
  バックテストしたくなった場合は自前実装か別ライブラリの検討が必要。
- **`Buy & Hold Return [%]` は戦略間で比較不可**: Backtesting.py はインジケータのウォームアップ後の
  バーを起点に B&H を計算するため、SMA75 を持つベースライン（75 バー後起点）とシグナル戦略（初日起点）で
  値が異なる。参考値として扱うこと。
- **J-Quants 無料枠の 12 週間遅延**（既知）: 直近 3 ヶ月のシグナル検証はこのデータではできない。
  直近分は yfinance 補完の構成（PoC-1 の結論）をシグナル runner にも適用可能（data_loader を差し替えるだけ）。
- 285A0（キオクシア）は期間中に約 15 倍に上昇しており、合算成績を支配する。戦略評価の際は銘柄別に見ること。
